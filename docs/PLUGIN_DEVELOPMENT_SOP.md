# 浏览器插件产品开发 SOP

本 SOP 适用于新建或迭代浏览器插件产品，从想法确认、产品设计、技术选型、开发验证到发布和后续维护。它不绑定某个具体产品、页面结构或技术栈；执行时应根据目标浏览器、用户场景和风险等级裁剪。目标是用最短路径获得可安装、可验证、可回退、可持续迭代的稳定版本。

## 0. 核心工作原则

1. **先对齐产品，再写代码**：先明确用户场景、状态和验收标准，避免实现后反复推翻。
2. **先前端闭环，再考虑后端**：本地能力能完成的 MVP，不提前引入账号、同步和服务器。
3. **一次只解决一个明确问题**：功能、修复和重构尽量分开，方便测试与回退。
4. **`main` 始终可安装**：未验证的代码不直接作为正式版本发布。
5. **权限最小化**：每项浏览器权限必须有对应功能和面向用户的解释。
6. **用户数据向后兼容**：更新版本不能无提示清空用户配置、收藏、任务或其他持久数据。
7. **视觉反馈不能牺牲效率**：动效和声音短、轻、可关闭，并尊重系统无障碍设置。
8. **每个版本可追溯**：需求、代码、版本号、CHANGELOG、ZIP 和 GitHub Release 保持一致。

## 1. 全流程总览

| 阶段 | 核心问题 | 必须产出 | 通过条件 |
| --- | --- | --- | --- |
| 1. 需求收敛 | 为谁解决什么问题 | 需求卡、范围和验收标准 | 无关键歧义 |
| 2. 产品结构 | 页面和状态怎样组织 | 信息架构、交互规则 | 主流程能闭环 |
| 3. 草图与视觉 | 页面长什么样 | 3 版草图、1 版视觉基准 | 方向明确确认 |
| 4. 技术方案 | 用什么能力实现 | 架构、权限、数据结构 | 风险可接受 |
| 5. 增量开发 | 如何最小成本交付 | 可安装代码、演示状态 | 单个需求完成 |
| 6. 分层验证 | 是否稳定可用 | 静态、交互、扩展环境测试 | 验收项通过 |
| 7. 试用反馈 | 真实使用是否顺手 | 问题清单和优先级 | 无阻断问题 |
| 8. 版本发布 | 如何可靠交付 | 版本号、CHANGELOG、ZIP、Release | 可下载可安装 |
| 9. 维护迭代 | 如何持续更新 | Issue、短分支、小版本 | main 始终稳定 |

## 2. 阶段一：需求收敛

### 2.1 每个需求先填写需求卡

```text
需求名称：
目标用户：
使用场景：
当前痛点：
期望结果：
必须包含：
明确不做：
验收标准：
是否影响数据：是 / 否
是否新增权限：是 / 否
风险等级：低 / 中 / 高
计划版本：
```

### 2.2 Definition of Ready

满足以下条件后才进入设计或开发：

- 能用一句话说明用户价值。
- 主操作对象和状态清楚。
- 明确正常、空、失败和边界状态。
- 有可观察、可测试的验收标准。
- 知道是否会改变权限、数据结构或隐私说明。
- 已记录本次不处理的范围。

### 2.3 需求优先级

- **P0**：数据丢失、无法安装、核心功能不可用、安全或隐私问题。
- **P1**：明显影响主流程，例如无法切换或关闭标签页。
- **P2**：体验缺陷，例如 tooltip 裁切、声音过小、布局异常。
- **P3**：锦上添花，例如更多动画、主题或个性化设置。

先处理 P0/P1，再处理高频 P2。不要为了 P3 延迟稳定版本。

## 3. 阶段二：产品结构和交互

### 3.1 先定义信息架构

在做视觉前，先确认：

- 页面有哪些业务区块。
- 区块顺序和层级。
- 单项、网站级和全部操作分别放在哪里。
- 每种操作执行后的状态变化。
- 哪些数据必须在标签页关闭后继续保留。
- 撤销能恢复什么，不能恢复什么。

推荐先画出与具体视觉无关的结构，例如：

```text
入口（工具栏图标 / 快捷键 / 右键菜单）
    ↓
核心工作区
    ├── 主要对象与状态
    ├── 单项操作
    └── 批量操作
    ↓
设置、帮助与异常恢复
```

是否使用 Popup、Side Panel、独立页面、页面内浮层或多入口组合，应由任务持续时间、所需空间和使用频率决定，不能直接套用其他产品的布局。

### 3.2 再定义状态模型

每项功能至少检查：

- 初始状态
- Hover / Focus / Active
- 操作进行中
- 操作成功
- 操作失败
- 空状态
- 数据很多时
- 页面已关闭时
- 重复添加时
- Undo 后

### 3.3 文案与本地化规则

- 在需求阶段确定产品 UI 的目标语言和是否支持多语言。
- 用户或第三方内容默认保持原始语言，除非产品明确提供翻译能力。
- 同一对象、状态和动作使用统一词汇，禁止同义词随意混用。
- 按目标语言的真实产品语境审校，避免逐字直译。
- 危险操作使用明确动词，并写清作用范围和不可恢复后果。
- 产品文案与内容文案分开管理，为未来本地化保留空间。

## 4. 阶段三：草图与视觉确认

### 4.1 三版草图法

正式 UI 前固定输出三种方向，但三版使用相同信息架构，避免同时比较结构和风格：

1. 轻盈编辑感
2. 深色空间工作台
3. 高密度效率工具

每版都必须使用真实长度的标题、摘要和多个网站卡片，禁止只用理想化短文本。

### 4.2 视觉定稿需要确认

- 桌面端还是弹窗端。
- 区块是纵向还是横向。
- 默认主题和系统主题策略。
- 卡片密度、列数和最小宽度。
- 长文本截断方式。
- 操作按钮、tooltip 和危险状态。
- 空状态和大量数据状态。
- 动效、声音和减少动态效果策略。

确认后把结论写入 `docs/DESIGN_SYSTEM.md`，设计结论不只保留在聊天记录中。

## 5. 阶段四：技术方案与风险评审

### 5.1 先选择产品载体

| 能力 | 适用场景 |
| --- | --- |
| Toolbar Popup | 高频、短时、少量操作 |
| Side Panel | 需要与当前网页并行工作的持续任务 |
| Full-page Dashboard | 信息密度高、需要大空间或长时管理 |
| Content Script UI | 必须直接叠加或操作当前网页内容 |
| Context Menu | 与选中文本、链接或图片相关的快捷动作 |
| Options Page | 低频配置、权限和账户设置 |
| Background Service Worker | 事件监听、状态协调和后台 API 调用 |

先选择最小可闭环组合，不默认创建所有页面。

### 5.2 再选择技术栈

- 根据目标平台选择 Chromium Manifest V3、Firefox WebExtensions 或 Safari Web Extension。
- 小型插件优先考虑原生 HTML / CSS / JavaScript，减少构建和依赖成本。
- 组件多、状态复杂或多人协作时，再评估框架、类型系统和构建工具。
- 只有跨设备、多人共享、账号、计费或服务端计算确有必要时才引入后端。
- 浏览器 API 应集中封装，避免平台差异散落在业务代码中。
- 必须为无法直接运行扩展 API 的本地预览提供演示数据或适配层。

### 5.3 权限评审门槛

新增或扩大权限前必须回答：

1. 哪个用户功能必须依赖该权限？
2. 能否使用更窄的权限或按需授权？
3. 用户会看到什么浏览器警告？
4. README 和隐私说明怎样解释？
5. 旧用户升级是否需要重新授权？

未回答清楚，不合入权限变化。

### 5.4 数据方案

- 先区分实时数据、会话数据、本地持久数据和云端数据。
- 可以从浏览器或页面重新获得的数据，不重复长期持久化。
- 用户明确创建、收藏或配置的数据，选择与产品承诺一致的存储位置。
- 新增字段必须提供默认值。
- 删除或重命名字段必须提供迁移。
- 正式引入复杂迁移前增加 `schemaVersion`。
- 永远不要把“清除扩展数据”当作正常升级方案。

### 5.5 风险等级决定验证深度

- **低风险**：README、注释、非功能文案，只检查链接和格式。
- **中风险**：样式、动效、声音、非核心交互，需要浏览器视觉和交互验证。
- **高风险**：权限、存储、关闭标签页、批量操作、后台服务，需要完整回归和旧版本升级测试。

## 6. 阶段五：Git 与开发执行

### 6.1 分支策略

`main` 只保留已验证版本。每个需求使用短生命周期分支：

```bash
git switch main
git pull --ff-only
git switch -c feat/export-data
```

常用前缀：

- `feat/`：新功能
- `fix/`：问题修复
- `docs/`：文档
- `refactor/`：不改变功能的重构
- `release/`：发布准备

### 6.2 实现顺序

1. 更新需求或规格文档。
2. 定位最少需要改动的文件。
3. 先实现数据和状态，再实现视觉反馈。
4. 为新增数据提供兼容默认值。
5. 加载扩展执行当前需求的最短验证。
6. 检查相邻功能，防止局部修复造成回归。
7. 更新测试清单和 CHANGELOG。

### 6.3 提交规则

一个提交只表达一个完整变化。推荐：

```text
feat: add JSON export for saved pages
fix: prevent tooltips from being clipped
docs: translate README to English
refactor: centralize storage access
```

提交前运行：

```bash
git diff --check
git status --short
```

不要提交 Token、账号信息、真实浏览记录、`.DS_Store`、临时截图或测试输出。

## 7. 阶段六：分层验证

验证顺序从快到慢，前一层失败时先修复，不急着进入后一层。

### 7.1 第一层：静态检查

```bash
node --check path/to/ui-script.js
node --check path/to/background-script.js
node --check path/to/content-script.js
python3 -m json.tool manifest.json >/dev/null
git diff --check
```

### 7.2 第二层：本地页面验证

使用演示数据检查：

- 桌面布局
- 长标题和摘要
- 目标断点和响应式布局
- Light / Dark / System
- tooltip 完整显示
- 折叠、搜索、确认弹窗
- 动效和光标效果
- 控制台无错误

本地页面验证只证明 UI 和普通 JavaScript 工作，不能替代真实扩展环境。

### 7.3 第三层：真实扩展验证

在目标浏览器重新加载插件，并测试：

- 所有声明入口均能打开正确界面或执行正确动作。
- 核心浏览器 API 在真实权限下工作。
- 关键对象的创建、读取、修改和删除正确。
- 持久数据在关闭页面、重启浏览器或升级后仍符合产品承诺。
- 批量操作、确认和 Undo 行为符合说明。
- 首次用户手势触发的音频、剪贴板或其他受限能力正常。
- Content Script 修改后，在刷新目标网页后正确生效。
- 受保护页面和权限不足时有合理降级。

### 7.4 第四层：回归矩阵

中高风险版本至少覆盖：

| 维度 | 状态 |
| --- | --- |
| 浏览器 | 产品声明支持的浏览器和最低版本 |
| 主题 | System、Light、Dark |
| 数据 | 空、少量、大量、重复 URL |
| 上下文 | 单窗口、多窗口、普通/无痕（若支持） |
| 页面 | 普通网页、受保护页面、无权限页面 |
| 辅助设置 | 产品设置组合、Reduce Motion、高对比度 |
| 操作方式 | 鼠标、键盘 |
| 升级 | 旧版本数据升级到新版本 |

完整清单见 `docs/RELEASE_CHECKLIST.md`。

## 8. Bug 修复 SOP

1. **收集证据**：截图、操作路径、浏览器、版本、主题、窗口宽度。
2. **稳定复现**：明确每次是否出现，找到最小复现条件。
3. **定位层级**：产品规则、HTML、CSS、页面逻辑、浏览器 API、权限还是数据。
4. **解释根因**：用一句话说明为何发生。
5. **最小修复**：只改必要范围，不顺便大规模重构。
6. **验证原问题**：使用相同条件重新测试。
7. **验证邻近场景**：例如修复最右 tooltip 后检查其他按钮、相邻卡片和两种主题。
8. **补充防回归项**：把问题加入 `RELEASE_CHECKLIST.md`。
9. **更新 Patch 版本**：用户可感知的正式修复发布修订版本。

可复用的典型经验：

- tooltip 显示不全要检查祖先元素 `overflow`、层级、弹出方向和最右对齐。
- 首次声音不播放要检查浏览器自动播放限制，并在用户手势阶段初始化 AudioContext。
- 视觉图方向错误要先确认目标设备和页面结构，再生成风格方案。

## 9. 版本号与发布判断

### 9.1 版本号

- **Docs only**：只改 README 或维护文档，不影响插件，不必更新 Manifest 版本。
- **Patch `x.y.Z`**：Bug、样式、文案、性能或小体验调整。
- **Minor `x.Y.0`**：新增向后兼容功能。
- **Major `X.0.0`**：不兼容的数据、权限或架构变化。

### 9.2 Definition of Done

- 需求验收标准全部通过。
- 静态检查通过。
- 与风险等级匹配的浏览器验证完成。
- 没有新增控制台错误。
- 权限和隐私说明准确。
- 旧数据兼容或有迁移方案。
- 版本号和 CHANGELOG 一致。
- README 安装说明仍然有效。
- ZIP 检查通过。
- 有明确回退版本。

## 10. 打包和 GitHub 发布

### 10.1 发布前提交

```bash
git add .
git commit -m "release: prepare vX.Y.Z"
git push -u origin <branch-name>
```

通过检查后合并到 `main`：

```bash
git switch main
git pull --ff-only
```

推荐通过 GitHub Pull Request 合并。个人快速迭代也应至少确认 `git diff` 和测试清单。

### 10.2 生成安装包

在仓库上一级目录执行，将变量替换为实际项目名：

```bash
PROJECT_DIR="your-extension"
PACKAGE_NAME="your-extension-vX.Y.Z.zip"
zip -rFS "$PACKAGE_NAME" "$PROJECT_DIR" -x '*.DS_Store' -x "$PROJECT_DIR/.git/*"
unzip -t "$PACKAGE_NAME"
```

确认 ZIP 解压后存在包含 `manifest.json` 的插件文件夹。

### 10.3 GitHub Release

- Tag：`vX.Y.Z`
- Title：`<Product Name> vX.Y.Z`
- Release notes：Highlights、Changes、Fixes、Install、Known limitations
- Asset：上传经过检查的版本 ZIP
- 勾选 Latest release
- 稳定版不勾选 Pre-release

### 10.4 发布后冒烟测试

1. 从 GitHub Release 下载刚发布的 ZIP。
2. 解压到新的临时目录。
3. 使用 Load unpacked 加载该目录，而不是开发目录。
4. 测试启动、核心主流程、持久化和一个高风险操作。
5. 确认 README 下载链接指向最新 Release。

只有从公开产物重新安装成功，才算真正发布完成。

## 11. 回滚 SOP

发现严重问题时：

1. 暂停继续传播有问题的 Release，并在说明中标记已知问题。
2. 确认最后一个稳定 tag 和 ZIP。
3. 不对已发布的 `main` 使用强制推送。
4. 使用 `git revert <commit>` 创建可追溯的反向提交。
5. 修复后发布新的 Patch 版本，不覆盖旧版本号。
6. 用旧数据重新执行升级测试。
7. 在 CHANGELOG 记录问题、影响范围和修复方式。

## 12. 高频迭代的最短执行卡

普通需求或 Bug 可以按下面 12 步快速执行：

1. 写一句用户问题。
2. 写 3–8 条验收标准。
3. 标注风险等级和版本类型。
4. 从最新 main 创建短分支。
5. 修改最小文件范围。
6. 运行静态检查。
7. 在本地页面验证视觉和普通交互。
8. Reload 扩展，验证真实浏览器 API、权限和持久化。
9. 执行相关回归项。
10. 更新 CHANGELOG、版本号和文档。
11. 提交、合并、打包并发布 Release。
12. 下载 Release ZIP 做一次全新安装冒烟测试。

任何一步失败，回到对应阶段解决，不带着已知问题继续向后发布。

## 13. 每月维护建议

- 检查 GitHub Issues 和用户反馈，合并重复问题。
- 检查所有目标浏览器的扩展政策、Manifest 和 API 变化。
- 检查权限是否仍然必要。
- 测试最新浏览器版本。
- 清理无效临时元数据，但不删除用户收藏。
- 检查 README、下载链接、版本号和 Release 是否一致。
- 保留至少一个上一稳定版本 ZIP。
- 按实际反馈更新本 SOP，而不是让流程与项目脱节。
